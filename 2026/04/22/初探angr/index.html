初探Angr

- Reverse

这篇笔记不是“命令速查表”，而是把 angr 真正常用的知识串起来：
为什么它能解题、对象之间是什么关系、什么时候该符号化、什么时候该 hook、什么时候该先跑到关键点再补约束。

适用方向：CTF Reverse、部分 Pwn、自动化求解校验逻辑。

1. angr 到底在干什么

很多人第一次接触 angr，会把它理解成“自动跑出 flag 的工具”。
这种理解太浅了

angr 的核心其实是：

  • 符号执行（Symbolic Execution）
  • 约束求解（Constraint Solving）
  • 程序状态建模（Program State Modeling）

它不是在“瞎跑”，而是在模拟程序执行时，把一部分输入设成符号变量，然后在每条路径上不断收集约束，最后交给求解器算出满足条件的输入。

所以 angr 的本质不是“爆破”，而是：

把程序逻辑翻译成数学约束，再反推出输入。

2. solver 是 angr 背后的大脑

  • angr 背后主要原理就是 solver
  • Z3 也是其中一种

在 angr 里，真正负责“算答案”的不是执行器，而是 solver。
常见底层求解器就是 Z3。

你可以把 angr 看成两层：

上层：程序语义模拟

负责：

  • 模拟寄存器
  • 模拟内存
  • 模拟栈
  • 模拟文件系统
  • 管理执行路径

下层：约束求解

负责：

  • 判断某个路径是否可达
  • 判断某组条件是否矛盾
  • 求出符号变量的具体值

所以 angr 并不是“只会跑程序”，它更像是：

执行引擎 + 数学求解器的组合体

也正因为如此，angr 非常适合处理：

  • 输入校验
  • 多分支迷宫
  • 算法校验函数
  • 某些字符串变换与比较
  • 可以转成位运算约束的问题

3. angr.Project：整个程序的抽象入口

我对 Project 的理解是“整个程序的抽象”

Project 负责什么

它不是单纯“打开一个文件”，而是把整个二进制程序包装成 angr 可以分析的对象。它负责：

  • 加载 ELF / PE / Mach-O
  • 识别架构（x86 / x64 / ARM 等）
  • 解析段、节、符号、入口点
  • 初始化 loader
  • 提供 factory 来创建各种状态对象

4. state：程序某一时刻的完整快照

我对angr本质的理解

  • state = 程序某一时刻的完整快照
  • 里面有寄存器、内存、栈、stdin/stdout、solver、执行历史

一个 state 里有什么

一个 state 可以理解成“程序在这一瞬间的世界全貌”，至少包括：

  • 寄存器状态：eax / ebx / rsp / rip ...
  • 内存状态：某地址里现在存了什么
  • 栈状态：栈帧如何组织
  • 文件描述符状态：stdin / stdout / stderr
  • 路径约束：当前路径要求满足哪些条件
  • 执行历史：从哪走到哪
  • 插件信息：如 posix、memory、solver、globals 等

这意味着：

angr 不是“只记住当前地址”，而是把程序执行环境整个建模出来。

常见创建方式

4.1 entry_state()

从程序正常入口开始。

适合：

  • 想完整模拟程序启动流程
  • 程序逻辑不复杂
  • 输入从 stdin 开始读

4.2 blank_state(addr=...)

从你指定的地址直接开始。

适合：

  • 你只关心某个函数
  • 前面初始化过程太麻烦
  • 想手动布置寄存器和内存

4.3 call_state()

直接构造一次“调用函数”的状态。

适合：

  • 单独分析某个 check 函数
  • 想按函数调用约定传参
  • 不想自己搭栈

为什么 blank_state 很常见

很多题如果从入口跑：

  • 初始化太长
  • 有无关逻辑
  • 有反调试
  • 有复杂 IO

这时就会直接从 check 函数前面起步，然后自己把输入塞到内存、寄存器或栈里。

8. claripy：符号表达式层

  • claripy.BVS('pass1', 32)
  • BVS 是未知位向量变量。fileciteturn0file0

这部分一定要真正理解，不然 angr 很容易变成“照抄脚本但不明白”

8.1 为什么是位向量，不是普通整数

二进制程序里很多运算都和位宽强相关：

  • 8 位字符
  • 32 位寄存器
  • 64 位地址
  • 移位
  • 按位与/异或
  • 溢出

所以 angr 不用普通 Python int 来描述符号值，而是用 bitvector。

8.2 BVS：Symbolic BitVector

表示：

  • 名字叫 x
  • 位宽 32
  • 值未知

8.3 BVV：Concrete BitVector

表示：

  • 一个具体常量
  • 数值为 1
  • 位宽 32

区分一下 BVS 和 BVV，这个点非常重要

一句话概括：

  • BVS：未知量
  • BVV：已知常量

8.4 常见约束写法

注意：

这里不是在做普通 Python 判断，而是在构造符号约束表达式

9. 把符号变量塞进程序：寄存器、栈、内存

angr 解题的核心动作之一，就是：

把“未知输入”放到程序会读取的地方。

这个地方可能是：

  • 寄存器
  • 栈
  • 某个缓冲区
  • 某个全局变量
  • 文件内容

9.1 塞进寄存器

意思就是：

  • 把 eax 设成一个 32 位符号变量
  • 后续程序如果读 eax，读到的就是“未知输入”

这适合：

  • 从某函数中途开始
  • 参数在寄存器中传递（x64 很常见）

9.2 塞进内存

表示把符号变量写进某个固定地址。

适合：

  • .data / .bss / 堆上某缓冲区
  • 程序后面会从这个地址读数据

9.3 读出内存内容再加约束

这个套路的意思是：

  1. 先让程序跑到关键检查点
  2. 在那个时刻读出将要参与比较的内存
  3. 再手动要求它等于目标值
  4. 最后反求原始输入

这在逆向题里特别好用，因为：

  • 你不一定需要把整个 check 跑完
  • 只要跑到关键比较前就够了
  • 后面复杂运算可以自己接管

9.4 直接符号化内存 vs 间接符号化内存

区分两个概念对理解angr非常有帮助

直接符号化内存

表示某地址里直接放输入内容。

间接符号化内存

先放一个指针，再在指针指向的区域里放真正内容：

这种情况常见于：

  • char *argv[]
  • char **
  • 指针数组
  • 多级引用

9.5 endness 为什么重要

写入地址值时必须考虑大小端

因为当你往内存里写的是“地址”“整数”“指针”，而不是普通字节串时，机器需要按架构字节序存放。

这个写法本质上是在说：

按当前程序架构的内存字节序去解释和写入这个值。

场景一：标准 stdin 校验题

适合特征：

  • 程序读 stdin
  • 跳 success/fail 分支清楚
  • 没有太复杂的外部依赖

基本思路：

场景二：只想分析 check 函数

适合特征：

  • 已经逆出来 check 函数入口
  • 不想从 main 开始跑
  • 参数位置清楚

思路：

  1. blank_state(addr=func_addr) 或 call_state()
  2. 手工构造参数
  3. 符号化输入
  4. 跑到关键比较点
  5. 取值或补约束

场景三：先跑到关键点，再手动补约束

适合特征：

  • 后半段逻辑太复杂
  • 你已经知道目标比较值
  • 只想让 angr 帮你“倒推”输入

思路：

  1. explore(find=check_addr)
  2. memory.load() 读出待比较内容
  3. add_constraints(check_bvs == desired)
  4. solver.eval(...)

场景四：复杂 check / 循环 / 加密逻辑

适合特征：

  • 原始机器码执行太慢
  • 有反调试
  • 有大量无关计算

思路：

  1. 找到关键函数或关键代码段
  2. 用 hook 或 SimProcedure 重写
  3. 仅保留关键语义
  4. 再让 angr 走路径

15. 初学 angr 时需要知道的事

15.1 angr 不是全自动神器

很多题并不是“写 10 行脚本就秒了”。
真正效率高的 angr 用法通常是：

  • 先用 IDA/Ghidra 弄清大致流程
  • 确定从哪里切入最省事
  • 只对关键输入做符号化
  • 尽量减少无关路径

也就是说：

angr 最强的时候，不是替代静态分析，而是配合静态分析。

15.2 不是所有变量都该符号化

如果你把：

  • 所有输入
  • 所有内存
  • 所有文件内容

全符号化，路径很容易爆炸。

更好的策略是：

只符号化真正影响最终校验的那部分。

15.3 “跑到目标点”往往比“全自动到 success”更实用

这点值得再强调一遍。

在很多逆向题里：

  • 直接找 success 太难
  • 但找到比较前一行很容易

这时先到关键点，再自己补约束，往往是最稳的打法。

15.4 hook 不是作弊，而是语义抽象

很多新手会觉得：

“我 hook 了，不就不是 angr 真跑出来的吗？”

其实不是。

真正的核心不是“坚持执行原机器码”，而是：

把你已经知道的程序语义告诉 angr，让它把精力花在你还不知道的输入约束上。

这正是符号执行在工程上真正有价值的地方。

  • Project 是程序抽象
  • state 是执行快照
  • simgr 管路径
  • explore 找目标
  • solver 算约束
  • claripy 造符号变量
  • hook / SimProcedure 改写程序语义。

所以 angr 最核心的一句话可以总结成：

把程序执行过程建模成状态迁移，再把成功路径对应的输入条件交给求解器反解。

如果只把它当成“自动跑 flag 的工具”，你会很快卡住。
如果把它当成“程序语义建模 + 约束求解框架”，你就会开始真正会用 angr

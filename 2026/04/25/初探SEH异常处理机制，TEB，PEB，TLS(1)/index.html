初探SEH异常处理机制,TLS,PEB,TEB原理

发表于2026-04-25|更新于2026-04-25

初探可能不是很详细，只是为了了解

SEH

定义:
SEH是微软自己的异常处理机制，是C/C++的Microsoft扩展，用于处理访问冲突，除0等异常情况，在C++里推荐ISO C++ exception
了解异常处理机制：
异常处理大致我们需要了解三层：
1.硬件/OS异常：访问冲突，非法指令，除零，这些异常先有CPU/OS产生
2.SEH:Windows提供的统一用户态异常分发框架，用来把这些异常一路送到对应处理器
3.C++ EH:编译器在Windows上实现C++ throw/catch时，可能借助或映射到SEH框架之上，但是不“等于”SEH

TEB

定义：
TEB（Thread Environment Block）描述线程状态
里面有一个指向PEB的指针，还有TlsSlots[64]和TlsExpansionSlots等TLS相关成员
理解：
- TEB是“线程级环境块”
- 每个线程都有一个
- 很多线程私有运行时状态都能沿着TEB找到
- x86常见通过FS段寄存器访问，x64常见通过GS寄存器访问
- 当线程创建时。loader会把线程TEB的地址放到x86的FS或x64所关联的位置；
- TLS数组与TEB有对应关系
在Windows逆向中的应用：
在逆向过程中总是可以看到
- fs:[0]、fs:[18h]、fs:[30h]
- x64下的gs:[…]
这些本质上都是从当前线程环境块拿数据

PEB

定义：

PEB（Process Environment Block）包含进程信息，例如：

  • BeingDebugged
  • Ldr：模块加载器数据
  • ProcessParameters：命令行等启动参数
  • SessionId 等

TEB 中有 ProcessEnvironmentBlock 指针，所以线程可以经由 TEB 找到所属进程的 PEB。官方 TEB 文档直接写了 ProcessEnvironmentBlock 字段就是指向 PEB 的指针

PEB一般用于：

  • 从 PEB 的 BeingDebugged 做反调试
  • 从 Ldr 遍历模块链表，不调用 API 也能找到 kernel32 / ntdll
  • 从 ProcessParameters 拿命令行、环境信息

PEB/TEB 是 Windows 内部结构

TLS

TLS (Thread Local Storage)，线程局部存储。它的重点不是“异常处理”，而是“每个线程有自己独立的一份数据”。

官方 TEB 文档说 TlsSlots[64] 和 TlsExpansionSlots 就是 TLS 数据区，正常访问应通过 TlsGetValue 之类 API。

官方 PE 格式文档则从镜像角度讲了 静态 TLS：

  • 链接器会生成 TLS Directory
  • loader 给模块分配 TLS 索引
  • 每个线程都有一份 TLS 数组
  • 程序通过 TLS 索引找到属于本模块的 TLS 数据区
  • .tls 目录还能登记 TLS Callback Functions，在线程/进程初始化和终止时调用，典型用途之一就是构造/析构对象

TLS 回调会在 main / WinMain 之前执行，所以：

  • 很多样本会把反调试、解密、环境初始化放到 TLS 回调
  • 你动态调试时发现“还没到入口点程序就已经做过事了”，常常就是 TLS 回调在先动手
